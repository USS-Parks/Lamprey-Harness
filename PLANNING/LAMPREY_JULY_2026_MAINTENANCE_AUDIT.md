**Date:** 2026-07-02 · **Version audited:** v0.15.6 (`main` @ `173c7f2`) · **Status: REPORT FOR REVIEW — no code changed.**

Six parallel read-only audit passes were run over the whole repo: (1) provider/chat core, (2) loop/automations/agentic infra, (3) main-process security surface, (4) renderer, (5) data layer, (6) config/CI/tests/docs/dependency currency. ## 0. Executive summary

**Measured baselines (2026-07-02):**

| Gate | CLAUDE.md claim | Actual |
|---|---|---|

**Headline defects, in order of what I'd fix first:**

1. **The Loop feature is broken at its core (LP-1, P0).** The iteration prompt built by `buildIterationPrompt` never reaches the model — `runHeadlessTurn` uses `promptBody` only for a hook and builds API messages from persisted history, and the loop controller (unlike `fireDueWakeups`) never persists the prompt. Every loop iteration runs a turn in which the model has no instruction, no current task, no ledger.
2. **Loops can run away even when the master toggle is OFF (LP-4/LP-5/LP-6, P0/P1).** `schedule_wakeup` is always registered, approval-free, checks `loopsEnabled` nowhere, has zero ceilings, and since LP-1 each fired wake-up runs a real turn — a self-perpetuating autonomous loop on a default install. Flipping the toggle off also doesn't stop already-running loops, and `loop_control continue` re-arms paused loops.
3. **Nothing serializes concurrent turns (LP-2, CC-8, P0).** Loop ticks re-fire a loop whose previous iteration is still running; wake-up bursts after Windows sleep can launch up to 50 parallel headless turns; and `activeAbortControllers` is keyed by conversationId with unconditional set/delete, so a loop turn and a user turn in the same conversation clobber each other's cancel handles and interleave stream events in one transcript.
4. **A pre-stream provider failure hangs the turn forever (CC-1, P1).** A throw before streaming starts (e.g. missing API key) never settles the `runChatRound` promise — spinner forever, ghost-reply guard never runs, abort controller leaks.
5. **Two remote-content P1s in the Electron shell (SEC-1, SEC-2).** No `will-navigate` guard means any top-frame navigation hands the full privileged `window.api` to the destination origin; and `setWindowOpenHandler` forwards `file://`/UNC/custom-scheme URLs straight to `shell.openExternal`.
6. **A crash mid-write can wipe all credentials or all settings (DB-1, DB-2, P1).** `keys.json` and `settings.json` are written non-atomically by five independent writers, and both readers silently "heal" corrupt JSON to `{}` — the next write persists the empty object.
7. **v0.15.6 ships with debug tracing force-enabled (SEC-3, P2)** — `forceDebugTraceOn()` in `main.ts` persists tool arguments to a plaintext 20 MB JSONL log despite its own "remove before shipping" comment.
8. **Currency debt (HY-2, HY-3, P1).** Electron 35 is ~8 majors / ~a year of Chromium CVEs behind and its pin reason no longer holds; the legacy `@xenova/transformers` 2.x carries the entire critical/high `npm audit` chain and is superseded by `@huggingface/transformers` 3.x.

**What was checked and is genuinely fine:** SQL injection (all stores parameterized — verified across every dynamic-SQL site); path/command confinement on `apply_patch` + `shell_command` (no traversal, `shell:false` everywhere); SSRF guards in `url-safety.ts` (loopback/link-local/RFC1918/redirect re-validation, wired into web tools + research extractor); keychain crypto handling (safeStorage, 0o600, no key values in audit events); preload exposes a curated API with no raw `ipcRenderer`; all windows/views run `contextIsolation:true, sandbox:true, nodeIntegration:false`; the T1 stream watchdog's timer/listener hygiene; the migration ledger's per-version transactions + downgrade refusal; README fully current for v0.15.6; CI action versions current; no `.only` in tests; no `rehype-raw` (model HTML is not rendered); Shiki's `dangerouslySetInnerHTML` is escape-safe.

**Total: 4 P0 · 21 P1 · 30 P2 · ~35 P3** after merging duplicates.

## 1. P0 — broken feature / runaway risk (all in the Loop layer)

**LP-1 · P0 · Loop iteration prompt never reaches the model.**
`electron/services/loop-controller.ts:410-435` passes the iteration prompt as `promptBody`, but `electron/ipc/chat.ts:525` uses `promptBody` only for the `promptSubmit` hook; API messages come from persisted history alone (`chat.ts:547,620`), and unlike `fireDueWakeups` (`loop-runner.ts:170-175`) the controller never calls `saveMessage`. On a fresh `/loop` conversation the model receives an empty history. The doc comment at `chat.ts:490` claiming apiMessages include the iteration prompt is false.
**Fix:** persist the iteration prompt as a user message before invoking the runner (mirror `fireDueWakeups`), or make `runHeadlessTurn` append `promptBody` to `apiMessages`. Add an integration test that asserts the prompt text appears in the messages actually sent.

**LP-2 · P0 · Concurrent duplicate iterations of the same loop.**
`startLoopController` (`loop-controller.ts:494-501`) ticks every 30s with no in-flight guard; `nextFireAt` advances only *after* the turn completes (`:340-345`), so `listDueLoops` (`loop-store.ts:214-221`) keeps returning a loop mid-iteration. A 5-minute turn ⇒ ~10 overlapping iterations, each computing `iteration + 1` and `tokensUsed` from the same stale row — duplicated spend, lost-update on the counter, `maxIterations` under-enforced.
**Fix:** mark the loop in-flight (advance `nextFireAt` or set a running flag/in-memory set) *before* awaiting `runTurn`; skip in-flight loops in `tickLoops`.

**LP-3 · P0 · The per-iteration stall watchdog is disconnected in production.**
`runLoopIteration` passes `signal: watchdog.signal` (`loop-controller.ts:273`) but `productionDeps().runTurn` (`:410-417`) drops it, `LoopTurnRunner` (`loop-runner.ts:37-41`) has no signal field, and the `chat.ts:464-470` wiring never forwards one — though `runHeadlessTurn` accepts `signal` (`chat.ts:516`). A hung provider stream never settles the await at `:269`: run row stuck `'running'`, backlog item stuck `'in_progress'`, and (via LP-2) each tick piles on another hung turn.
**Fix:** thread the signal through `LoopTurnRunner` → `runHeadlessTurn`, and make the watchdog reject via `Promise.race` as a backstop.

**LP-4 · P0 · `schedule_wakeup` bypasses `loopsEnabled` with zero ceilings.**
Registered always-enabled, `requiresApproval: false` (`loop-tool-pack.ts:11-60`), in the default `'full'` surface; `scheduleWakeup`/`fireDueWakeups` (`loop-runner.ts:82-207`), `loops:schedule` IPC (`ipc/loops.ts:26-43`), and `startLoopWakeups()` (`main.ts:612`) check `loopsEnabled` nowhere. Since LP-1's seam, each fired wake-up runs a real headless turn (`loop-runner.ts:187-197`) — so on a default install (loops OFF) the model can schedule a wake-up whose turn schedules the next: a self-perpetuating loop with no iteration/wall-clock/token cap.
**Fix:** gate `scheduleWakeup` + `fireDueWakeups`' turn execution on `readLoopConfig().enabled`; add a per-conversation pending-wakeup cap; consider `requiresApproval: true` for the tool.

## 2. Loop / automations layer — P1/P2/P3 (`LP-*` continued)

**LP-5 · P1 ·** Master toggle OFF doesn't stop running loops — `tickLoops` (`loop-controller.ts:457-462`) never checks `.enabled`; only create/resume are gated (`ipc/loops.ts:87-90,156-158`), which is exactly (and only) what `loop-safety.test.ts` locks. **Fix:** early-return from `tickLoops` when disabled; widen the safety test.
**LP-6 · P1 ·** `loop_control 'continue'` (`loop-tool-logic.ts:88-94`) re-arms paused loops regardless of `loopsEnabled` and regardless of *user* pause intent. **Fix:** check config + refuse continue on user-paused loops.
**LP-7 · P1 ·** Automations dedup key drops the date: `automations-runner.ts:210-211` dedups on `Number(\`${hours}${minutes}\`)` while the correct date-qualified `minuteKey` (line 199) is discarded (`void minuteKey`, line 215) — a daily `0 9 * * *` automation fires once per app session; HHMM collisions (1:23 vs 12:03 → 123) falsely suppress. **Fix:** use the already-computed `minuteKey` string.
**LP-8 · P1 ·** Wake-up burst after system sleep: `fireDueWakeups` (`loop-runner.ts:159-207`) selects all due LIMIT 50 and launches each fire-and-forget — after an overnight sleep, dozens of parallel LLM calls on the first tick. **Fix:** run sequentially or cap in-flight; collapse wake-ups stale by more than one interval.
**LP-9 · P1 ·** *(merged with CC-8)* `activeAbortControllers` keyed by conversationId with unconditional set/delete (`chat.ts:158,637-641,661-663`): a loop/wake-up turn clobbers a live user turn's registration; `chat:cancel` aborts the wrong one; whichever finishes first deletes the shared entry; `chat:chunk` events from both turns interleave. **Fix:** refuse/queue a second turn per conversation, or key by correlationId with per-conversation cancel fan-out.
**LP-10 · P1 ·** Workflow sandbox poisons host `Math.random`: `workflow-runner.ts:691` passes the *host* `Math` into the vm; the preamble's `Math.random = __randomBlock` (`:733`) mutates it process-wide — after one workflow run, every later `Math.random()` in the main process throws until restart. **Fix:** pass a frozen copy/proxy.
**LP-11 · P1 ·** Max-wallclock counts idle/paused time: `started_at` set once at creation (`loop-store.ts:160-162`), never reset on resume; vs default 30-min `maxWallclockMs` (`loop-config.ts:21`) — `/loop 1h <task>` dies `max-wallclock` at its second pre-flight. **Fix:** track cumulative active time, or reset on resume + document semantics.
**LP-12 · P2 ·** No crash recovery: `in_progress` backlog items + `'running'` run rows are never swept at startup (`loop-controller.ts:231-238`, `loop-store.ts:336-343`) — task silently never retried; loop can end `'backlog-empty'` with work frozen. **Fix:** startup sweep demoting stale `in_progress` → `pending`, closing orphaned runs. *(Also DB-audit finding.)*
**LP-13 · P2 ·** Per-iteration state transitions non-transactional (`loop-controller.ts:277-345`): crash between `finishLoopRun` / `updateBacklogItem` / `updateLoop` leaves contradictory audit state and skewed ceiling accounting. **Fix:** wrap the commit phase in `db.transaction`.
**LP-14 · P2 ·** Cron automations have **no master toggle at all**: `startAutomations()` runs unconditionally (`main.ts:611`) and executes any enabled automation via `chatOnce` (`automations-runner.ts:200-214`). **Fix:** gate behind `loopsEnabled` (or its own flag).
**LP-15 · P2 ·** Stop/pause doesn't abort the in-flight turn (`ipc/loops.ts:146-174`) — status flips in DB, but no abort signal reaches `runHeadlessTurn` (same root as LP-3). **Fix:** loopId-keyed AbortController map, aborted from stop/pause.
**LP-16 · P2 ·** No per-automation in-flight guard: an every-minute automation whose `chatOnce` takes >60s accumulates unbounded concurrent runs of itself (`automations-runner.ts:214`). **Fix:** skip if previous `runOne` still pending.
**LP-17 · P2 ·** *(merged with DB finding)* Ceiling semantics of `0` inconsistent: `tokenBudget=0` disables, `maxIterations=0`/`maxWallclockMs=0` instant-stop (`loop-controller.ts:53-66`, `loop-config.ts:27-29`) — violates the app-wide "0 disables" convention from the Timeouts phase. **Fix:** normalize 0-means-off for all three (map 0 → null) or reject 0.
**LP-18 · P2 ·** Error/timeout iterations skip post-ceiling checks and token accounting (`loop-controller.ts:348-381`) — consistently-erroring loops spend real tokens never counted against `tokenBudget`. **Fix:** best-effort estimate on error + run post-flight ceilings there too.
**LP-19 · P3 ·** `loops:create` accepts non-finite `intervalSeconds` (`ipc/loops.ts:113`); NaN → NULL `next_fire_at` → treated as always-due, firing every 30s tick. **Fix:** `Number.isFinite` at the IPC boundary.
**LP-20 · P3 ·** `reorderBacklog` renumbers blindly (`loop-store.ts:408-415`) — stale renderer list ⇒ duplicate positions, arbitrary `nextBacklogItem` order. **Fix:** reorder only pending; re-append unlisted after.
**LP-21 · P3 ·** No FKs/existence checks on `loop_backlog`/`loop_runs` (`loop-schema.ts:33-59`); `loops:enqueue` (`ipc/loops.ts:192-199`) inserts orphans for deleted/typo'd loopIds. **Fix:** validate `getLoop(loopId)` or add FKs.
**LP-22 · P3 ·** Model's `continue` delay honored only for self-paced loops (`loop-controller.ts:336-339`) — autonomous-loop backoff requests are clobbered by `computeNextFire`. **Fix:** honor a future `nextFireAt` for all modes.
**LP-23 · P3 ·** Automations cron uses local wall-time, skips missed minutes, no DST handling (`automations-runner.ts:74-82,222-226`) — a daily 02:30 automation never fires on spring-forward day. **Fix:** document, or compare against a rolling window.
**LP-24 · P3 ·** Quit doesn't drain in-flight loop/wake-up turns before `closeDb()` (`main.ts:676-680`) — noisy shutdown errors + more orphaned `'running'` rows. **Fix:** track in-flight promises; race against a short drain timeout on quit.

## 3. Provider / chat core — `CC-*`

**CC-1 · P1 · Pre-stream throw hangs the turn forever.** `chat.ts:762-1078` wraps `chatStream(...)` in `new Promise` without attaching `.catch` to `chatStream`'s returned promise; `getClientForProvider` (`registry.ts:409-411`) throws before the retry loop, so neither `onDone` nor `onError` fires. Missing/deleted API key ⇒ `chat:send` never returns, spinner forever, ghost-guard never runs, abort controller leaks. **Fix:** `.catch` → persist/emit error + `reject`, or hoist client/key resolution ahead of the wrapper.
**CC-2 · P1 · Retry appends onto stale accumulators.** `registry.ts:765-767` — `fullContent`/`fullReasoning`/`toolCallsAccumulator` aren't reset between retry attempts (`:1103-1109`, `:1151-1163`): a mid-stream stall + retry persists `partial₁ + full₂` (duplicated prefix) and concatenates attempt-2 argument fragments onto attempt-1's partial tool call — malformed JSON that then silently parses to `{}` (CC-6). **Fix:** reset all three at the top of each retry iteration.
**CC-3 · P1 · `onDone` body unprotected.** `chat.ts:784-999` — a throw from `saveMessage` (`:866,890,956`), `addMemory` (`:1312`), or a rejected approval in the `Promise.all` (`:918`) is an unhandled rejection on a floating promise (`registry.ts:1051` fires `onDone` without awaiting) — turn never settles, no error row, no ghost notice (the v0.9.2 class of failure). **Fix:** wrap the whole `onDone` body in try/catch routed to `reject` (the pattern already at `:997-999`).
**CC-4 · P1 · Fallback tool-calling is dead code.** `FALLBACK_TOOL_INSTRUCTION` (`fallback-tool-parser.ts:89-99`) has zero injection sites — no branch of `buildSystemPrompt` mentions the `{"action":...,"input":{...}}` contract, so `supportsTools:false` models (qwen3.5-plus/flash, qwen-long) and FC-10-downgraded models are *expected* to emit a format they were never told about; a downgrade silently means "tools stop working." **Fix:** inject the instruction + rendered tool list when `!supportsTools || isDowngraded`, and rebuild the prompt on mid-conversation downgrade.
**CC-5 · P1 · Persisted `role:'system'` rows never reach the model.** `chat-history.ts:114` drops them, breaking the documented R1+R2 research fallback (`chat.ts:310-327`) whose comment claims the "answer from training ONLY" note is "picked up via promptHistory" — the model answers as if its search succeeded. **Fix:** map persisted system rows into the API array, or stop claiming they reach the model.
**CC-6 · P1 · Malformed native tool args silently coerced to `{}`.** `chat.ts:1097-1101` (`catch { args = {} }`) — tools without `required` fields sail past `detectEmptyParams` and FC-5 validation and execute with defaults instead of returning a corrective tool result; combined with CC-2, a mutating tool can run with dropped parameters. **Fix:** short-circuit with an `argument_parse_failed` tool result echoing the raw string.
**CC-7 · P1 · Custom models never consulted by `resolveModel`.** `ipc/model.ts:52-74` saves `provider` etc., but `resolveModel` (`registry.ts:424-446`) falls back to a hard-coded DeepSeek descriptor for any unknown id — the catalog's own comments (`registry.ts:186-188,299-301`) tell users to paste DashScope/AI-Studio ids into Custom Models, which then dispatch to `api.deepseek.com` with the DeepSeek key. **Fix:** merge `settings.customModels` before the fallback.
**CC-8 · P1 ·** = LP-9 (abort-controller clobbering), listed there.
**CC-9 · P2 ·** Ghost-reply guard only lives in the `chat:send` catch (`chat.ts:390-407`); failed wake-up/loop turns are only `console.error`'d (`loop-runner.ts:194-196`) — the exact G1/D5 ghost class. **Fix:** move the guard into `runHeadlessTurn`'s catch/finally.
**CC-10 · P2 ·** Reasoning-trail byte cap enforced with a character slice (`reasoning-trail.ts:47-54`) — CJK trails can persist ~3× the 64 KB cap and split surrogate pairs. **Fix:** `Buffer`-aware truncation.
**CC-11 · P2 ·** Loop token-budget estimate undercounts by up to an order of magnitude on tool-heavy turns (`chat.ts:624-628,658-660`): counts the initial stack once, string content only; multi-round turns re-send the growing stack every round; CJK ~2-3× more. CLAUDE.md's "slightly undercount" is materially wrong. **Fix:** accumulate per-round sent-chars inside `runChatRound` and return the sum; correct the doc.
**CC-12 · P2 ·** Per-conversation cleanup never wired: `clearToolUnlockState` / `resetCapabilityTracking` (`tool-unlock-state.ts:17-20,68-73`; `capability-tracker.ts:41,52-61`) have zero production callers — maps grow unbounded and an FC-10 downgrade pins a conversation to fallback mode until app restart. **Fix:** call from conversation-delete IPC.
**CC-13 · P2 ·** Fallback-JSON validation failure returns `null` and the raw JSON blob is rendered as the assistant's visible reply (`fallback-tool-parser.ts:161-166` + `chat.ts:832-849`); the in-file comment describes a feedback contract no caller implements. **Fix:** structured validation-failure result fed back as a synthetic tool round.
**CC-14 · P2 ·** Streaming tool-call accumulator keys on `tc.index` with no `undefined` guard (`registry.ts:1007-1019`) — a loose OpenAI-compat shim (Google's has history here) merges all parallel calls into one corrupt entry. **Fix:** `idx = tc.index ?? accumulator.size`, new entry on new `id`.
**CC-15 · P2 ·** `reasoning_content` echo gating inconsistent: in-turn it's sent unconditionally for all providers (`chat.ts:900-905`), cross-turn it's gated to exact ids `deepseek-v4-pro|flash` (`chat-history.ts:21-26`) — strict compat layers can 400; legacy `deepseek-chat` rows miss the v0.15.4 fix. **Fix:** gate both sites on `resolveModel` output.
**CC-16 · P3 ·** `research/synthesizer.ts:46` still defaults to retired `'deepseek-v3'` — works only via `RETIRED_MODEL_MAP`. **Fix:** change to `deepseek-v4-flash`.
**CC-17 · P3 ·** The two `toolSurface` gates disagree (`chat.ts:592` "anything but `'full'`" vs `:679` "exactly `'lazy'`") — a typo'd value yields full catalog + stubbed skill bodies. **Fix:** normalize once, share the boolean.
**CC-18 · P3 ·** `toolResultSpillBytes` measures UTF-16 chars, not bytes (`chat.ts:947-953`, `tool-result-spill.ts:27-28,79`). **Fix:** rename or measure with `Buffer.byteLength`.
**CC-19 · P3 ·** `isUserAbortError` matches any message containing "abort" (`ghost-reply-guard.ts:48-52`) — "connection aborted by upstream" is misclassified as a user cancel and the ghost-guard skips. **Fix:** match `err.name === 'AbortError'` + known cancel strings only.
**CC-20 · P3 ·** `chat:send` catch dereferences `err.message` on untyped throws → `{success:false, error: undefined}` violates the IPC contract (`chat.ts:363-367,408`); research/normal success payload shapes also differ (`:306` vs `:362`). **Fix:** `err instanceof Error ? err.message : String(err)`; unify payloads.
**CC-21 · P3 ·** Dead/vestigial: `system-prompt-builder.ts:326,433` still runs `.replace(PSEUDO_TAG_GUARD,'')` (guaranteed no-op scan every prompt build since L6); `services/deepseek.ts` legacy shim; `validateViaChatProbe` (`registry.ts:493-495`) claims "cheapest model" but takes the *first* (V4 Pro). **Fix:** delete/correct.
**CC-22 · P3 ·** settings.json parsed by five independent readers with divergent defaults; `chat.ts:947` re-reads the file every tool round. **Fix:** one cached accessor with mtime invalidation. *(Pairs with DB-2.)*
**CC-23 · P3 ·** `chat:generateTitle`'s `chatOnce` has no abort/timeout beyond the SDK's ~10-min default (`chat.ts:437-455`). **Fix:** `AbortSignal.timeout`.

## 4. Security surface — `SEC-*`

**SEC-1 · P1 · No navigation guard on the main window.** `main.ts:316-357`; zero `will-navigate`/`will-attach-webview` handlers in `electron/`, and no `ipcMain.handle` validates `senderFrame`. Any top-frame navigation (e.g. a crafted link in rendered model/research output) hands the destination origin the full privileged `window.api` (file read, MCP spawn, shell). **Fix:** `will-navigate` → `preventDefault` for non-app URLs + assert sender origin in privileged handlers.
**SEC-2 · P1 · `setWindowOpenHandler` forwards any scheme to `shell.openExternal`.** `main.ts:328-331` — `file://`, custom-protocol, or `\\attacker\share` UNC URLs pass unfiltered (contrast the correctly-guarded IPC at `:448-452`); a click launches a local program or leaks NTLM creds to an SMB host. **Fix:** allow `http(s)` only, mirroring the IPC handler.
**SEC-3 · P2 · Debug tracing force-enabled in the shipped build.** `main.ts:544` calls `forceDebugTraceOn()` unconditionally (its own comment says remove before shipping); `debug-trace.ts:33-36,112-126` appends tool args to plaintext `userData/lamprey-debug.log` (20 MB + `.prev`, never GC'd); `mcp-manager.ts:463` logs `argsPreview` of every MCP call. **Fix:** delete the call so tracing honors `settings.debugTrace`.
**SEC-4 · P2 · File-read IPC unconfined.** `files:readText` (`ipc/files.ts:156`), `:listDir` (`:144`), `:walkProject` (`:181`) read any absolute path — chained with SEC-1 (or any renderer compromise) this reads `~/.ssh/id_rsa` or `userData/keys.json`. **Fix:** confine to workspace root (reuse `resolveCwdWithinWorkspace`) or gate behind approval.
**SEC-5 · P2 · MCP stdio servers spawn arbitrary processes with no approval.** `mcp:addServer` (`ipc/mcp.ts:79`, sanitizer `:8-38` validates only the id) + plugin `connectors.json` (`mcp-manager.ts:200-282`) + the bundled `npx @anthropic-ai/mcp-server-playwright` auto-running at startup (`:117-121`, fetch-and-execute of a remote npm package). **Fix:** route stdio-server creation through the approval flow; pin/verify the bundled package.
**SEC-6 · P2 · Auto-update unsigned.** `electron-builder.yml:80` (`signAndEditExecutable:false`), `updater.ts:22-23` (`autoDownload` + `autoInstallOnAppQuit`) — integrity rests solely on `latest.yml`'s sha512 from the update host; anyone who can alter the manifest pushes a trojaned auto-installed update. **Fix:** code-sign the Windows build (known cost/effort tradeoff — decision item).
**SEC-7 · P3 · `.bucket.json` is tracked in git** (committed in `7ec30f3`; the `.gitignore:23` rule is dead for already-tracked files) — leaks CF account/zone IDs + bucket/CDN names (recon only; live creds are elsewhere and ignored). **Fix:** `git rm --cached .bucket.json`; decide separately on history scrubbing (IDs already public in history). *(Also found by hygiene pass.)*
**SEC-8 · P3 · Artifact sandbox CSP lacks `navigate-to`/`form-action`** and neither artifact surface sets a deny-all `setWindowOpenHandler` (`artifact-sandbox.ts:16,91-105,146-160`) — artifact HTML can exfiltrate its own contents via `location=`. Defense-in-depth only. **Fix:** extend CSP + deny window-open on the artifact view/window.

**Verified fine:** apply-patch/shell confinement, SSRF guards (documented DNS-rebind TOCTOU is honest), keychain crypto + OAuth CSRF `state`, curated preload, sandbox flags everywhere.

## 5. Renderer — `RD-*`

**RD-1 · P1 · Stuck streaming state on conversation switch.** `chat-store.ts:238-260` resets `toolCalls`/`runPhase` but not `isStreaming`/`streamingContent`; `useChat.ts:82-92` drops terminal events for non-active conversations — switch mid-stream ⇒ A's partial text renders in B, `isStreaming` stays true forever, `canSend` (`ChatInput.tsx:1124`) locks the input app-wide, Stop cancels the wrong conversation. **Fix:** clear streaming state in `selectConversation`; let terminal events reset `isStreaming` regardless of `matchesActive` (or track the streaming conversation id).
**RD-2 · P1 · `/fork` + session-duplicate read a field that doesn't exist.** `ChatInput.tsx:704` and `sessions-store.ts:223` read `res.data.id`, but `conversation:fork` returns `{conversationId}` (`ipc/conversation.ts:466`) — `/fork` navigates to `undefined`; Duplicate never navigates. (`chat-store.ts:314` does it correctly.) **Fix:** read `conversationId` at both sites.
**RD-3 · P1 · No staleness guard on `selectConversation`'s async load.** `chat-store.ts:242-252` applies `getMessages` unconditionally — rapid A→B switching can leave B active rendering A's messages. **Fix:** bail if `get().activeConversationId !== id` after the await.
**RD-4 · P2 ·** Concurrent approval requests overwrite each other; the replaced one dies to the 30s auto-deny unseen (`App.tsx:256-269`, `ToolApprovalModal.tsx:17,39-48`). **Fix:** queue modal requests like `inline-approvals-store` queues chips.
